PR Summary

The old route structure used many route files as thin wrappers around private page files in matching dash-prefixed folders. The new structure keeps the main page code beside the route declaration, so opening a route file answers both "what URL is this?" and "what does it render?"

Inline Page Entry Points

Page components from one-page private folders were moved into their matching route files, including top-level routes, application forms, cruise routes, and the root layout.

Support Code Stayed Local

Schemas moved into segment-local -schemas/ folders. Root error and not-found handlers remain in -root/ because they are support components, not route pages.

Behavior Preserved

Route URLs and IDs stayed the same. routeTree.gen.ts had no meaningful route metadata change after generation and build.

What Changed

Route files now own their default render

Files such as login.tsx, applications/$applicationId/formA.tsx, and cruises/new.tsx now contain their page component directly instead of importing it from a one-off sibling folder.

Route API usage is closer to the route

Same-route calls that previously used getRouteApi('/path') were replaced with Route.useSearch() or Route.useParams() where that route already has the generated Route object in scope.

Private folders are now support folders

Dash-prefixed folders are reserved for route-local support code: schemas, dense extracted pieces, and root-level error or not-found components.

Before And After

The biggest difference is navigational. Reviewers can now start at a route file and see the route contract plus the primary page component without bouncing through a matching private page folder.

Areas Touched

Area Changed Shape Reviewer Focus
Top-level routes accountsettings, confirmemail, cruiseapproval, cruiseeffects, forgotpassword, help, index, login, mypublications, newcruise, priorityinformation, register, resetpassword, and usermanagement now keep the rendered page component in the route file. Confirm route guards, search validation, imports, and page rendering remained equivalent.
Applications applications/index.tsx, applications/$applicationId/index.tsx, and formA.tsx, formB.tsx, formC.tsx now own their page components. Form schemas live in applications/$applicationId/-schemas/. Form behavior, validation schema imports, and application-specific params are the main things to scan.
Cruises cruises/index.tsx, cruises/new.tsx, and cruises/$cruiseId/index.tsx now contain their default page components. The cruise form schema lives in cruises/-schemas/. Confirm create/edit cruise flows still use the same route params, validation, and query behavior.
Root route __root.tsx now includes RootLayout. -root/ remains for app-wide error and page-not-found handlers. Confirm auth provider wiring, router outlet placement, and error/not-found behavior are unchanged.

Compatibility Notes

  • No route file locations changed.
  • No route URL, route ID, or intended navigation behavior changed.
  • Existing guards, loaders, search validation, and params are preserved.
  • TanStack Query hooks remain inside route-local or child components where that was already the natural ownership boundary.
  • Playwright page-object classes under frontend/tests/fixtures/pages/ were not renamed; they are test support objects, not React route page components.

Reviewer Notes

Good first files

Start with login.tsx, applications/$applicationId/formA.tsx, cruises/new.tsx, and __root.tsx. They show the refactor pattern clearly.

Most diff is relocation

The important review question is whether imports, route hook usage, and schema paths still line up after moving page code into the route file.

Support folders still matter

-schemas/ and -root/ remain intentional local support folders. The refactor removes one-page route shadow folders, not every dash-prefixed folder.

Out Of Scope

  • No UI redesign or copy changes were intended.
  • No auth, validation, query, or mutation behavior was redesigned.
  • No route URLs, route IDs, or file route paths were changed.
  • No Playwright page-object class rename was included.
